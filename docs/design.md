
# 桌面日历工具开发参考 Guideline

## 1. 项目定位

本项目是一个**个人向、本地化、轻量级桌面日历工具**。
目标不是做完整任务管理软件，也不是做云同步日程系统，而是复刻“桌面日历”类软件最核心、最顺手的那一部分体验：

* 在桌面右侧长期显示月历
* 按日期记录简单事项
* 支持事项完成/未完成
* 本地存储
* 不引入会员、同步、复杂配置等额外系统

项目技术路线固定为：

* **Tauri**
* 前端负责显示、编辑、交互
* 后端负责数据存储、转译、设置持久化、窗口行为控制

---

## 2. 核心设计原则

### 2.1 极简优先

一切设计以“够用、顺手、低维护”为优先，不为未来可能需求提前复杂化。

### 2.2 数据结构轻量化

不建立复杂 task 对象系统，不引入排序、标签、优先级、提醒链路、同步状态等冗余字段。

### 2.3 交互接近原始桌面日历体验

用户主要在月视图中完成查看与轻量编辑，而不是进入独立页面做任务管理。

### 2.4 Markdown-like 仅用于内部存储

内部存储借用 `- [ ] task` / `- [x] task` 格式，但不要求用户直接编辑 markdown 语法。

### 2.5 前台简洁，后台转译

前端展示和编辑的是“逐行任务文本 + 完成状态”，后台统一转译成内部存储格式。

---

## 3. 功能边界

## 3.1 本期必须实现

### 日历显示

* 月视图显示
* 公历日期
* 星期标题
* 周次显示
* 节假日标签显示（静态数据）
* 支持上月 / 下月 / 回到今天

### 日期事项

* 点击某个日期查看/编辑该日事项
* 每日事项为多行文本，每行一条
* 支持完成/未完成
* 按创建顺序显示，不支持手动排序

### 桌面行为

* 固定在桌面右上区域
* 不允许拖拽移动
* 允许改窗口大小
* 窗口大小变化时，以右上角为锚点
* 常驻
* 托盘存在
* 支持显示/隐藏/退出
* 开机启动
* 本地存储

### 数据持久化

* 每日事项使用 SQLite 持久化
* 设置项本地保存

---

## 3.2 明确不做

以下功能不在当前范围内：

* 云同步
* 账号系统
* 会员功能
* 导出/导入
* 调休/补班规则
* 手动拖拽排序
* 标签、优先级、截止时间
* 富文本编辑
* 子任务系统
* 多端同步
* 完整提醒系统
* 拖拽移动窗口
* 复杂搜索

---

## 4. UI 与布局约束

## 4.1 主窗口布局

主窗口分为两层：

### 顶部栏

包含：

* 当前年月
* 上一页
* 下一页
* 回到今天
* 设置入口
* 托盘相关控制入口可选

### 月历主体

* 7 列网格
* 按月显示（以当前日期所在周为第二行，上扩展1周，下扩展3周）
* 可包含周次窄列
* 每个日期格显示：

  * 日期号
  * 节日标签（如有）
  * 当天事项

### 当前布局修正结论

基于参考界面，主窗口应保持“**月历为绝对主体，编辑仅在需要时浮出**”的策略，而不是长期占据一个固定编辑区。

因此本项目在布局上补充以下约束：

* 主窗口默认状态只呈现顶部栏 + 月历主体，不长期保留底部大编辑面板
* 月历网格应尽可能完整铺开，优先保证“看月历”而不是“看表单”
* 选中某一天后，编辑器以**悬浮小弹框**形式出现，叠加在月历之上
* 编辑弹框不应推动或压缩整个月历布局
* 同一时刻只允许一个日期进入展开编辑态
* 点击其他日期时，编辑弹框应切换锚点并复用同一套轻量编辑器
* 编辑框应从被点击日期格的原位置“长出来”，而不是像独立气泡一样漂浮在旁边

## 4.2 视觉与行为约束

* 整体风格深色半透明
* 周末可弱着色
* 今天必须有明显边框高亮
* 节假日以轻量标签呈现
* 不做复杂装饰动效
* 月历格内内容优先清晰而非花哨
* 日期选中高亮应克制但明确，避免大面积抢戏
* 编辑弹框视觉上应明显轻于主网格，避免形成第二个“主界面”

## 4.3 日期格内显示规则

每个日期格展示当天事项摘要：

* 自动按创建顺序显示
* 建议最多显示前 2～3 条
* 超出后显示 `+N`
* 已完成事项灰显
* 显示层自动生成序号，但**序号不写回存储**

示例显示：

```text
1. 流体力学实践作业
2. 多智能体学习
+2
```

## 4.4 默认尺寸与比例建议

当前版本补充一条明确约束：

* **默认窗口比例应为横向宽于纵向高**

原因：

* 月历是 7 列结构，横向空间不足时最先受损的是整体阅读体验
* 宽一些可以减少日期格拥挤，同时保留每格中 1～2 条摘要的可读性
* 弹出式编辑器出现后，宽屏布局仍能维持月历主体稳定，不会显得局促

建议默认尺寸方向：

* 默认宽高建议落在 `900~1100 x 680~760`
* 优先推荐从 `960 x 720` 一类的横向比例开始调试
* 最小尺寸依然要保证月历 7 列不塌陷

这条建议会覆盖当前“默认窗口偏竖向”的原型方向。

---

## 5. 编辑交互规范

## 5.1 编辑方式

点击某一日期后，在原地弹出一个小编辑窗口，不跳转页面，不打开大型独立弹窗。

编辑窗口职责：

* 显示当前日期
* 展示该日全部事项
* 支持输入与修改
* 支持完成状态切换
* 支持确认保存

补充更具体的交互方向：

* 编辑器应锚定在被点击日期格附近，而不是固定出现在页面底部
* 优先采用紧凑浮层卡片，而不是第二块长期占位的大面板
* 编辑器尺寸以“够编辑当天事项”为准，不追求承担整个应用的主布局
* 当天无内容时，也只显示一个轻量空编辑框，而不是展开大面积空白区域
* 进入编辑的手势优先为**双击日期格**
* 单击日期格只负责选中，不直接展开编辑
* 编辑态下点击其他空白区域，应自动保存并关闭编辑框

---

## 5.2 编辑模型

编辑窗口中的逻辑单位为“行”。

约定：

* 一行 = 一条任务
* 回车 = 新建下一行
* 空行 = 无效行，保存时忽略
* 不支持单条任务内换行
* 任务展示顺序 = 行顺序 = 创建顺序

推荐具体表现：

* 默认仍可保持“每行一条”的逻辑
* UI 上应采用接近“连续文本编辑”的轻量输入体验，而不是一排排厚重表单项
* 用户关注点应始终是“写这一日的几条事项”，而不是管理复杂控件
* 编辑时界面应自动显示行序号，如 `1.`、`2.`、`3.`
* 行序号由显示层自动生成，不写入数据库，不要求用户手动输入
* 回车后自动生成下一条带序号的新行
* 用户面对的应是“简单文本框 + 自动编号”，而不是复杂列表管理器

---

## 5.3 编辑框形态约束

参考目标效果，编辑弹框补充以下形态规范：

* 宽度应明显小于单周宽度总和，形成局部浮层，而非整页抽屉
* 头部只保留必要信息：当前日期 + 确认按钮
* 主体应是简洁的事项编辑区，不堆叠额外功能按钮
* 保存按钮应足够直接，避免出现一排与当前阶段无关的提醒、标签、扩展操作
* 弹框四周留白应充足，视觉感受应是“轻、薄、好落笔”
* 弹框出现后，不应遮挡过多无关日期区域
* 若日期位于边缘位置，弹框允许自动调整到左侧、右侧、上方或下方，保证不超出窗口边界
* 弹框内不放“提示文案区”“功能说明区”“帮助文字区”，避免占用垂直空间
* 弹框左上角应与原日期格的左上起点保持继承关系，形成“从原格向外展开”的观感

推荐结构应尽量接近：

* 顶部：当前日期
* 右上：确认保存按钮
* 主体：连续多行任务编辑区
* 每条任务前：一个完成状态按钮

明确不加入：

* 闹钟按钮
* 倒计时按钮
* 标签按钮
* 提醒入口
* 与当前阶段无关的扩展操作栏
* 占空间的说明文字

---

## 5.4 完成状态规则

完成状态不通过复杂对象管理，而通过转译规则完成。

推荐行为：

* 用户点击某条任务前的完成按钮，切换该行 done / undone 状态
* 前端将该行状态设为 done
* 保存时转译为 `- [x] 内容`
* 未完成则转译为 `- [ ] 内容`

可兼容行尾符号输入，但不应作为主逻辑。

补充约束：

* 完成按钮应直接贴近每条任务行首，减少横向和纵向浪费
* 不需要为一条任务提供多个操作按钮
* 完成按钮已经足够承担当前版本的“任务操作入口”

例如：

编辑态：

```text
流体力学实践作业
多智能体学习 ✓
最优控制作业2
```

存储态：

```md
- [ ] 流体力学实践作业
- [x] 多智能体学习
- [ ] 最优控制作业2
```

---

## 6. 数据模型约束

## 6.1 数据库原则

数据库只存**用户事项数据**，不存整套日历。

公历日期、星期、周次应通过运行时计算得到。
节假日标签使用静态本地 JSON（如果可能存储较大则可以舍弃节假日）

---

## 6.2 主表设计

建议当前阶段只保留一张主表：

```sql
CREATE TABLE daily_entries (
  date TEXT PRIMARY KEY,      -- YYYY-MM-DD
  raw_md TEXT NOT NULL,       -- 内部 markdown-like 存储
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
);
```

说明：

* `date` 作为唯一键
* `raw_md` 存储该日全部任务
* 一天没有任务时，可直接删除该行，不保留空记录

---

## 6.3 设置项存储

设置项不必进入 SQLite，建议单独存储为本地配置：

建议存储内容包括：

* 窗口宽度
* 窗口高度
* 是否开机启动
* 是否显示周次
* 是否显示节假日标签
* 上次查看月份
* 主题参数（如透明度）

---

## 7. 内部存储与转译规范

## 7.1 内部标准格式

内部统一使用 markdown checkbox 风格：

* 未完成：`- [ ] 内容`
* 已完成：`- [x] 内容`

例如：

```md
- [ ] 流体力学实践作业
- [x] 多智能体学习
- [ ] 最优控制作业2
```

---

## 7.2 读取规则

从 `raw_md` 读取时：

* 匹配 `- [ ] xxx` -> 未完成项
* 匹配 `- [x] xxx` -> 已完成项
* 其他非标准行 -> 尽量按普通未完成文本兼容读取

输出给前端的数据建议是“行模型数组”，而不是原始 markdown 字符串。

例如：

```json
[
  { "text": "流体力学实践作业", "done": false },
  { "text": "多智能体学习", "done": true },
  { "text": "最优控制作业2", "done": false }
]
```

---

## 7.3 保存规则

前端编辑完成后，将行模型重新序列化为 `raw_md`。

规则：

* 空白行忽略
* 保持原有顺序
* 已完成行序列化为 `- [x]`
* 未完成行序列化为 `- [ ]`
* 若最终无有效任务，则删除该日期记录

---

## 8. 前后端职责划分

## 8.1 前端职责

* 月历渲染
* 日期格展示
* 编辑窗口展示
* 行输入与回车行为
* 完成状态视觉反馈
* 调用后端接口读写数据
* 窗口大小相关布局更新

---

## 8.2 后端职责

* SQLite 存储
* `raw_md` 与行模型之间的转译
* 月份数据聚合
* 设置项持久化
* 托盘管理
* 开机启动
* 窗口行为控制

---

## 9. 推荐接口设计

建议后端只暴露最少量命令。

### 获取当前视图数据

`get_calendar(date)`

返回：

* 当前天对应日历格数据
* 每天事项摘要
* 节假日标签
* 周次信息

### 获取某日内容

`get_day_entry(date)`

返回该日期对应的行模型数组。

### 保存某日内容

`save_day_entry(date, lines)`

其中 `lines` 为前端行模型。

### 切换某行完成状态

`toggle_line_done(date, line_index)`

可选保留。
如果前端编辑框已经足够顺手，也可统一只走 `save_day_entry`。

### 设置读写

* `get_settings()`
* `save_settings(payload)`

---

## 10. 窗口行为规范

## 10.1 位置规则

窗口固定锚定在**桌面右上角**。

要求：

* 不支持拖拽改变位置
* 调整大小时右上角坐标不变
* 默认保留当前截图所示的桌面布局逻辑：右侧月历面板，左侧保留桌面空间

---

## 10.2 常驻规则

* 应长期显示
* 不应遮挡正常工作窗口
* 托盘必须可控制显示/隐藏
* 关闭主窗口时建议隐藏到托盘，而不是直接退出

---

## 10.3 尺寸规则

* 允许用户调整窗口大小
* 应设置最小宽高，避免月历网格塌陷
* 过小时优先保证日期和基本网格可读
* 事项显示不足时允许摘要裁切，不强行完整展示
* 默认窗口不应以偏高窄条形态启动，默认比例应更接近横向月历工作台
* 调整尺寸后，仍应优先保护月历主体的完整度，再考虑编辑浮层的位置补偿

---

## 10.4 当前原型修正意见

针对当前原型，明确以下修正方向：

* 当前“底部长期编辑区”不符合目标体验，应替换为“点击日期后出现的轻量悬浮编辑框”
* 当前默认窗口比例偏竖向，不利于月历主体展示，应改为横向优先
* 当前页面层级应进一步收敛，让用户第一眼看到的是月历，而不是编辑器
* 当前交互优化优先级高于继续追加桌面化功能实现
* 当前编辑器应从“多控件表单”继续收缩为“自动编号文本编辑 + 行首完成按钮”
* 当前原型中一切非必要说明文案和扩展按钮都应删除

---

## 11. 异常与边界处理

## 11.1 空内容

当天编辑后若无任何有效任务，保存时删除该日期记录。

## 11.2 重复任务

允许同一天存在内容完全相同的多条任务。
完成切换必须按**行索引**而不是按文本匹配。

## 11.3 非法格式兼容

若 `raw_md` 中存在非标准行，读取时尽量按普通文本兼容，不直接报错。

## 11.4 长文本

单条任务过长时：

* 月视图中单行省略
* 编辑框中可完整显示

## 11.5 连续回车

连续空行不应写入数据库。

---

## 12. 推荐开发顺序

## 第一阶段：最小可运行版本

目标：先做出完整闭环。

* 月视图静态渲染
* 日期点击
* 编辑弹窗
* 单日事项保存
* SQLite 存储
* 读取展示
* 完成状态切换

## 第二阶段：桌面化能力

* 托盘
* 开机启动
* 固定右上角
* 调整大小
* 设置项持久化

## 第三阶段：体验打磨

* 节假日标签
* 周次显示
* 今天高亮
* 月视图摘要优化
* 视觉主题统一

---

## 13. 实现时必须避免的方向

以下做法容易把项目复杂化，应避免：

* 过早引入 task_id、排序系统、标签系统
* 让用户直接编辑 markdown 语法
* 为未来同步需求设计复杂数据层
* 把日历基础数据塞进数据库
* 把“显示序号”写入存储内容
* 过早抽象成通用任务平台

---

## 14. 当前版本的一句话定义

本项目是一个：

**固定在桌面右上区域、可调大小、带托盘、本地 SQLite 存储、按日期编辑轻量事项、内部以 markdown checkbox 语法序列化的个人向 Tauri 桌面月历工具。**
